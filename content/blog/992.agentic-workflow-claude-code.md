Claude Code works out of the box. Drop into a repo, ask it to fix a bug, and it performs reasonably well. But "reasonably well" leaves a lot on the table. The difference between a generic AI assistant and one that genuinely feels like a development partner comes down to structure — not smarter prompts.

Over the past months I've iterated on a setup that turns Claude Code into something closer to a junior developer who has read the team's onboarding docs. The key insight: **structure your repo so the model understands your system naturally**, rather than trying to craft the perfect prompt every time.

## The Anatomy of My Setup

The configuration lives across several layers, each serving a different purpose:

| Layer | File | Scope |
|-------|------|-------|
| Global rules | `~/.claude/CLAUDE.md` | Every project |
| Project rules | `CLAUDE.md` (repo root) | This project only |
| Hooks | `~/.claude/hooks/*.sh` | Sound notifications |
| Settings | `~/.claude/settings.json` | Permissions + hooks wiring |
| Commands | `.claude/commands/*.md` | Project shortcuts |
| Memory | `~/.config/macrodata/` | Cross-session persistence |

These layers compose. Global rules set the baseline. Project rules add specifics. Commands provide shortcuts. Hooks enable parallel workflows. Memory provides continuity.

## CLAUDE.md — Onboarding Docs, Not Prompts

The most impactful file in the entire setup is `CLAUDE.md`. Think of it as onboarding documentation for a new team member, not as a prompt you're feeding to an AI.

A new developer joining your team needs to know: what tools do we use, what patterns do we follow, and what mistakes should they avoid? CLAUDE.md answers these same questions.

My global `~/.claude/CLAUDE.md` covers the stack-wide basics:

```markdown [~/.claude/CLAUDE.md]
# Global Development Preferences

## Package Manager & Runtime
- Always use **pnpm** (never npm or yarn)

## Frontend Stack
- **Nuxt 4** with Vue 3 Composition API (`<script setup lang="ts">`)
- **@nuxt/ui v4** for components and theming
- **Tailwind CSS v4** — use `@theme` blocks in CSS, not `tailwind.config.ts`
- **TypeScript** everywhere — strict mode preferred

## Code Style
- Prefer Tailwind utility classes over custom CSS
- Use auto-imports (Nuxt provides them for composables, components, utils)
- Prefer `const` over `let`, avoid `var`

## When Building
- Read existing code before modifying — follow established patterns
- Prefer editing existing files over creating new ones
- Don't over-engineer: solve what's asked, nothing more
```

The project-level `CLAUDE.md` adds specifics that only apply to this repo:

```markdown [CLAUDE.md]
# Project: vernaillen.dev

## Known Quirks
- `@shikijs/engine-javascript` and `@shikijs/engine-oniguruma` are explicit deps
  to work around pnpm strict hoisting on Vercel
- Server-side `queryCollection` must be imported from `@nuxt/content/server`
- `defineOgImage` component props go inside the `props` key, not at the top level
```

These quirks are the kind of tribal knowledge that takes a new developer weeks to discover. Writing them down saves Claude (and future you) from repeating the same mistakes.

## The "Do NOT" Section Changed Everything

The single most valuable addition to my CLAUDE.md was the "Do NOT" section. Positive instructions tell Claude what to do. Negative instructions prevent the mistakes it keeps making.

```markdown [~/.claude/CLAUDE.md]
## Do NOT
- Do NOT start dev servers, background processes, or watchers
- Do NOT install packages without asking — suggest the command instead
- Do NOT create new files when editing an existing one would work
- Do NOT add comments, docstrings, or type annotations to code you didn't change
- Do NOT use `npm` or `yarn` — always `pnpm`
- Do NOT use `tailwind.config.ts` — Tailwind v4 uses CSS `@theme` blocks
- Do NOT use `@nuxt/ui-pro` — use `@nuxt/ui` v4
- Do NOT generate barrel files (`index.ts` re-exports) unless one already exists
- Do NOT amend commits or force-push unless explicitly asked
- Do NOT add error handling for impossible scenarios
```

Every rule here exists because Claude made that mistake at least once. The dev server rule? Claude started `pnpm dev` in the background and then couldn't interact with it. The `@nuxt/ui-pro` rule? Claude kept suggesting imports from the old package. The barrel files rule? Claude loves creating `index.ts` files that re-export everything.

The project-level file gets its own "Do NOT" section for repo-specific traps:

```markdown [CLAUDE.md]
## Do NOT (Project-Specific)
- Do NOT use `queryCollection` from auto-imports in server routes
  — import from `@nuxt/content/server`
- Do NOT put `defineOgImage` props at top level — use the `props` key
- Do NOT add `@shikijs/*` packages — already explicit deps for Vercel
- Do NOT run `pnpm dev` — ask the user to start it
```

The compounding effect is real. After a few weeks of adding rules as mistakes happen, Claude rarely makes the same mistake twice. The "Do NOT" section is the closest thing to organizational memory that a stateless model can have.

## Hooks: Sound Notifications for Parallel Work

One of Claude Code's best features is that you can fire off a task and work on something else while it runs. The problem: you need to know when it finishes or when it needs your attention.

My setup uses three simple bash scripts that play macOS system sounds:

```bash [~/.claude/hooks/notification.sh]
#!/bin/bash
# Play sound + speak when Claude needs attention
afplay /System/Library/Sounds/Funk.aiff &
say "Your agent needs attention"
```

```bash [~/.claude/hooks/stop.sh]
#!/bin/bash
# Play sound + speak when Claude finishes
afplay /System/Library/Sounds/Glass.aiff &
say "Your agent has finished"
```

```bash [~/.claude/hooks/subagent_stop.sh]
#!/bin/bash
# Play sound when a subagent finishes
afplay /System/Library/Sounds/Pop.aiff
```

These get wired up in `~/.claude/settings.json`:

```json [~/.claude/settings.json]
{
  "hooks": {
    "Notification": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash ~/.claude/hooks/notification.sh",
            "timeout": 10,
            "async": true
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash ~/.claude/hooks/stop.sh",
            "timeout": 10,
            "async": true
          }
        ]
      }
    ]
  }
}
```

The `async: true` flag ensures the sound plays without blocking Claude's execution. The workflow becomes: start a task in one terminal, switch to another terminal or your browser, and listen for the chime. When you hear the glass sound, Claude is done. When you hear the funk sound, Claude needs a decision from you.

This small addition enables a genuinely parallel workflow where you can run multiple Claude instances across different projects simultaneously.

## Skills: Domain Knowledge on Demand

Claude Code supports skills — markdown files that provide domain-specific knowledge. The key principle is progressive disclosure: don't load everything at once, load it when relevant.

I have skills installed for my core stack:

- **nuxt** — Nuxt 4 patterns, server routes, composables
- **nuxt-ui** — Component props, theming, UI patterns
- **nuxt-content** — Collections, MDC rendering, queryCollection API
- **vue** — Composition API, props/emits, VueUse
- **document-writer** — Blog post writing style and structure

Skills activate automatically based on context. When I edit a `.vue` file, the Vue skill loads. When I write a blog post, the document-writer skill loads. Claude Code matches the current task to available skills without manual intervention.

You can also create your own skills. A skill is just a markdown file in `~/.claude/skills/` that describes conventions, patterns, or domain knowledge. The format is simple: a heading, a description of when to use it, and the actual content.

## Custom Commands: Project Shortcuts

Custom commands are markdown files in `.claude/commands/` that act as reusable prompts. They appear in Claude Code's autocomplete when you type `/`.

My project has two:

```markdown [.claude/commands/verify.md]
Run the full verification suite:
1. Run `pnpm lint:fix` and report any remaining errors
2. Run `pnpm typecheck` and report any type errors
3. Summarize: pass/fail for each step
```

```markdown [.claude/commands/new-blog-post.md]
Create a new blog post:
1. Check `content/blog/` for the next available number (lowest unused)
2. Ask for: title, description, estimated read time, slug
3. Create `content/blog/{number}.{slug}.md` with standard frontmatter
4. Create image dir at `public/images/blog/{number}.{slug}/`
5. Add introductory paragraph placeholder
```

The `/verify` command replaces the mental overhead of remembering which checks to run after every change. The `/new-blog-post` command encodes the project's blog post conventions so I don't have to explain them every time.

Commands are simple, but they compound. Every repeated workflow that you turn into a command saves you from re-explaining it in every conversation.

## Persistent Memory with Macrodata

Claude Code is stateless by default — each conversation starts fresh. The [Macrodata](https://github.com/toddmahood/macrodata) plugin adds persistent memory through a structured file system:

- **State files** — Current focus, active projects, communication preferences
- **Entity files** — Detailed notes on projects, people, topics
- **Journal** — Time-stamped observations and decisions

At the start of each session, Claude reads the state files and knows what I'm working on, what decisions were made previously, and what my preferences are. During the session, it updates these files as context changes.

The practical effect: I can close a conversation, open a new one days later, and Claude picks up where I left off without re-explaining the full context.

## Permissions: Start Tight, Grant Intentionally

Claude Code's permission system controls what the agent can do without asking. My `settings.json` starts with a curated allowlist:

```json [~/.claude/settings.json]
{
  "permissions": {
    "allow": [
      "Bash(git add:*)",
      "Bash(git commit:*)",
      "Bash(pnpm lint:*)",
      "Bash(pnpm lint:fix:*)",
      "Bash(pnpm build:*)",
      "Bash(gh issue list:*)",
      "WebSearch"
    ],
    "defaultMode": "default"
  }
}
```

The `defaultMode: "default"` means Claude asks permission for anything not in the allowlist. Over time, one-off approvals accumulate — random `WebFetch` domains, project-specific bash commands, debug helpers. I periodically review and clean these up, keeping only the universally useful entries.

A good rule of thumb: if a permission only makes sense for one project, move it to that project's `.claude/settings.local.json` instead of the global file.

## What I Learned

**Structure beats prompts.** A well-organized CLAUDE.md file outperforms clever prompt engineering every time. The model doesn't need creative instructions — it needs clear, structured context.

**The "Do NOT" section is the most valuable part.** Positive rules tell Claude what to do. Negative rules prevent the mistakes that waste your time. Every correction should become a rule.

**Keep hooks simple.** My first attempt used ~100-line TypeScript files. Five-line bash scripts do the same job. Use the simplest tool that works.

**Curate skills manually.** Install only what matches your actual stack. More skills means more context loaded, which means slower responses and potential conflicts.

**Clean up permissions regularly.** One-off grants accumulate fast. Review them monthly and remove anything that's no longer relevant.
